# PC互換チェッカー＆ゲーム推奨構成 修正指示書

## 背景

E2Eテスト8件（互換性チェッカー3件、ゲーム推奨構成5件）を実施した結果、
2つのコーナーそれぞれに品質問題があり、かつ両者が連携していないことが判明した。
以下は、テスト検証結果から導き出した修正指示の全体像である。

---

## 修正の全体方針

**原則：Claudeの判断を変えるのではなく、Claudeに渡すデータを変える。**

プロンプトに「〇〇を間違えるな」と追記するアプローチは確率的で信頼できない。
入力データを構造的に制約し、間違いが物理的に発生しない設計にすること。

---

## Phase 1：互換性チェッカーの品質確保（最優先）

### 1-1. user_specified の区別（最優先・工数中）

**問題：** ユーザーが型番で指定したパーツに対して不要な変更案が表示される。

**修正内容：**

- `_extract_parts_with_claude` の戻り値に `user_specified_parts` フィールドを追加する
- `_suggest_build_with_claude` のプロンプトに以下を注入する：

```
【変更禁止パーツ（ユーザー指定）】: RTX 4090 GAMING X TRIO, Z890 AORUS MASTER
これらのパーツは互換性NGであっても変更提案をせず、問題の説明のみ行うこと。
```

- フロントエンド側で `source: "user_specified"` のパーツには変更案ボタンを非表示にする

### 1-2. GPU/ケース変更案のトリガー条件（実装済みだが要確認）

**問題：** verdict の集計値だけで分岐していたため、RAM非互換でもGPU変更案が出ていた。

**確認事項：** 以下のマトリクスが正しく実装されているか検証すること。

| チェック結果 | GPU変更案 | ケース変更案 |
|---|---|---|
| GPU干渉マージン NG | ✅ 出す | ✅ 出す |
| フォームファクター不一致 | ❌ 出さない | ✅ 出す |
| ソケット不一致 | ❌ 出さない | ❌ 出さない |
| RAM非互換 | ❌ 出さない | ❌ 出さない |
| ケース未指定 + GPU干渉なし | ❌ 出さない | — |

**追加推奨：** 現在の正規表現ベースのitem判定は脆い。
check項目にcategoryフィールド（例：`gpu_clearance`, `socket`, `form_factor`）を
持たせることを検討すること。即時対応は不要だが、check項目が増えた際のリファクタ候補。

### 1-3. confirmed_parts の3状態管理（高・工数中）

**問題：** ターン間でパーツ情報が消える、または不整合が発生する。

**データ構造：**

```typescript
interface ConfirmedPart {
  name: string;
  category: string; // 'gpu' | 'cpu' | 'mb' | 'ram' | 'case' | 'psu' | 'cooler' | 'storage'
  source: 'user_specified' | 'ai_accepted' | 'ai_pending';
}
```

**状態遷移ルール：**

- ユーザーが型番を直接入力 → `user_specified`（変更提案しない）
- AIが推奨し、ユーザーが承認（「それで」「お願い」等）→ `ai_accepted`
- AIが推奨したが未反応 → `ai_pending`（次ターンで再提案可）

**暫定の承認判定：** NLPによる承認判定は工数大のため後回し。
代わりに以下のルールで暫定運用する。

- ユーザーが次の要望を出した（＝前の構成に異議なし）→ `ai_accepted` に昇格
- `ai_pending` のまま2ターン経過 → 自動的に `ai_accepted` に昇格

### 1-4. パーツDBの精度改善（中・工数小〜中）

**判明している誤り：**

- Z890 AORUS MASTER のフォームファクターが EATX と登録されている → 正しくは ATX
- GPU電源コネクタ情報に曖昧なフィールド名がある（MBのCPU電源と混同される）

**継続的なDB検証の仕組み：**

値域チェックスクリプトを作成し、スクレイピング後に自動実行する。

```python
# 検証ルール例
VALID_MB_FF = {'Mini-ITX', 'mATX', 'Micro-ATX', 'ATX', 'E-ATX'}
VALID_SOCKETS = {'LGA1851', 'LGA1700', 'LGA1200', 'AM5', 'AM4'}

# 製品名パターン → 期待フォームファクターのヒューリスティクス
FF_HEURISTICS = [
    (r'XTREME|GODLIKE|CREATION', 'E-ATX'),
    (r'MASTER|ELITE|PRO|PLUS|GAMING', 'ATX'),
    (r'ITX|MINI', 'Mini-ITX'),
    (r'mATX|MATX|MICRO', 'mATX'),
]
# パターンとDBの値が不一致の行を「疑いリスト」として出力 → 人間が確認
```

### 1-5. プロンプト改善（中・工数小）

以下を互換性チェッカーのシステムプロンプトに追加する。

- `mb_power_connector`（マザーボードのCPU補助電源）と `gpu_power_connector`（GPU補助電源）を
  明確に区別するフィールド名の指示
- フォームファクター判定時、DB値をそのまま使用し、推測しない指示
- ストレージカテゴリ（M.2スロット割り当て、SATAポート数、3.5インチベイ数）を
  互換性チェック項目に含める指示
- ユーザーが言及していない用途（VR等）を勝手に追加しない指示

## Phase 2：ゲーム推奨構成の品質改善

### 2-1. ゲーム推奨スペックDBの確認と修正（高・工数小）

**問題：** Tarkovの推奨メモリが64GBと表示される（実際の推奨は16GB）。

**対応：**

- ゲームタイトル別の推奨スペックデータを確認する
- DBにTarkov推奨64GBというデータが入っていないか探す
- DBにない場合、プロンプト内でClaudeが生成している → 推奨スペックはDB値のみを
  使用し、不明な場合は「公式推奨スペックを確認してください」と返す指示を追加

### 2-2. パーツスペックの正確な注入（高・工数小）

**問題：** Ryzen 9 9900Xが「24コア」と表示される（正しくは12コア24スレッド）。

**対応：**

- パーツDBからコア数・スレッド数を取得し、プロンプトに注入する
- Claudeにコア数を推測させない。DB値がない場合は「コア数不明」と表示する

### 2-3. 出力の再現性向上（中・工数小）

**問題：** 同じ質問（Tarkov/15万）で推奨CPUが 9800X3D と 9700X でブレる。

**対応：**

- API呼び出し時の temperature を 0.3〜0.5 に下げる
- 可能であれば、ゲームタイトル×予算帯のマッピングテーブルを用意し、
  推奨構成の骨格をDB側で固定する（Claudeには説明文生成のみ任せる）

### 2-4. 予算未指定時のデフォルト挙動（中・工数小）

**問題：** 予算未指定でいきなり40万超の構成が出る。

**対応：**

- 予算未指定時はミドル帯（15〜20万）をデフォルト提案とする
- 上位構成と廉価構成を1行ずつ添える（現在のテスト8の廉価代替案の形式で良い）

## Phase 3：2コーナーの連携

### 3-1. 推奨構成 → 互換性チェッカーへの引き渡し（Phase 1完了後に実施）

**問題：** 2つのコーナーが完全に独立しており、推奨構成で出たパーツの互換性が保証されない。

**修正内容：**

A. `/api/recommend` のレスポンスに `confirmed: false` を明示追加

B. 推奨構成カード内に「この構成で互換チェック →」ボタンを追加

```javascript
function transferToCompatCheck(buildItems) {
  // 「互換チェックへ引き継ぐ」ボタン押下 = 推奨構成の承認とみなす
  confirmedParts = buildItems.map(b => ({
    name: b.name,
    category: b.category,
    source: 'ai_accepted'
  }));
  switchMode();
  appendAIBubble(
    'ゲーム推奨構成を引き継ぎました。ケース・電源など追加パーツを指定するか、' +
    'このまま互換性チェックを実行できます。'
  );
}
```

C. `switchMode()` の `confirmedParts` 処理を明確化する
  - 引き渡しボタン経由 → `confirmedParts` を保持
  - ヘッダーの切り替えボタン経由 → `confirmedParts` をリセット
  （偶発的な状態漏れを防止する）

## 検証方法

各Phase完了後、以下のテストケースで再検証すること。

**互換性チェッカー用：**

- テストA：「ROG MAXIMUS Z890 HEROにRTX 4080 SUPERとCorsair 4000D Airflowで組みたい」
  → 3パーツとも変更案が出ないこと、Z890にIntel CPUのみ推奨されること

- テストB：「NZXT H1 V2にRTX 4090 GAMING X TRIOを入れたい」
  → GPU干渉NGが検出され、GPU変更案とケース変更案の両方が出ること

**ゲーム推奨構成用：**

- テストC：「Escape from Tarkovを高設定で安定60fps出したい。予算は15万くらい」
  → Tarkov推奨メモリが64GBと表示されないこと、再実行で大きくブレないこと

- テストD：「4K動画編集とAfter Effectsメインで使いたい。できるだけ静かなPCがいい」
  → 64GB RAM推奨、コア数表記が正確であること

- テストE：「ゲームもするし配信もしたい。あとAIの画像生成にも興味ある」
  → ユーザーが言及していない用途が追加されないこと、VRAM重視の構成であること

**連携用（Phase 3完了後）：**

- テストF：テストCの推奨構成 →「この構成で互換チェック」→ ケース指定
  → 推奨パーツが confirmed_parts に引き継がれ、互換性チェックが正常動作すること
